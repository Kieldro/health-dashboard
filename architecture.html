Request flow — what happens when you load the page

1. Browser → https://health.keo.life/ → CF edge → tunnel → localhost:8888 → serve.py sends index.html.

2. Page imports app.js (with cache-busting ?v=Date.now()), which imports data.js.

3. data.js fires 13 parallel fetch('/api/*') calls — each hits a handler in serve.py that runs one SQL statement against ~/HealthData/DBs/garmin.db (or reads a JSON master file for HR-recovery / zone-minutes).

4. All responses come back as JSON with Cache-Control: no-store; data.js reshapes them (e.g. computes Navy BF%, weekly mileage, moving averages).

5. app.js hands each dataset to a Chart.js canvas. The skeleton-pulse overlay on each card fades as the chart populates.

6. The page auto-reloads when you tab back in after 1h, or every 6h if left open.

Notable design choices

SQLite as the source of truth. The sync pipeline owns garmin.db; the dashboard only reads. Adding a chart = adding one SQL query in serve.py and one dataset in app.js.

No build step. Plain HTML + ES modules + Chart.js from a CDN. Edit a file → reload the page. The only "deployment" is systemctl --user restart health-dashboard.service.

Stateless server. Every API call is a fresh SQL query. The one exception is /api/zone-minutes, which caches its CPU-heavy result keyed on the input files' mtime.

Public via Cloudflare Tunnel. No open inbound ports on this machine; cloudflared dials out to CF edge and CF terminates TLS. The dashboard is reachable from anywhere without VPN, and the box itself stays firewalled.

Where this page gets its data: the SQLite-table chips fetch /api/schema, the route chips fetch /api/routes, the schedule reads /api/cron.
